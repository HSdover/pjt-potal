# Screen Domain Template

작성일: 2026-05-19

이 문서는 기능목록 엑셀을 기준으로 화면 도메인과 화면 템플릿 유형을 분류한 초안이다.
현재 단계에서는 화면의 형태와 분류만 정리하며, 기능 구현 방식은 다루지 않는다.

## 범위

포함:

- 기능목록 엑셀의 A01~G02 항목
- 상위 화면 도메인
- 화면 후보명
- 화면 템플릿 유형
- 통합/분리/보류 메모

제외:

- 화면 상세 항목
- API, 테이블, ERD
- 권한/인증 로직
- 외부 시스템 호출 방식
- 실패 처리, 큐, webhook
- 실제 프론트엔드/백엔드 구현

## 화면 도메인 요약

| 도메인 | 코드 | 설명 |
|---|---|---|
| 메타 관리 | A01~A03 | 메타데이터 검색, 상세 조회, 등록/수정 화면 후보 |
| AI 데이터 권한 관리 | B01~B02 | Agent와 데이터셋 매핑, 접근 신청 화면 후보 |
| 파이프라인 관리 | C01~C02 | 파이프라인 목록 조회, 상용배포 신청 화면 후보 |
| 신청서 승인 | D01~D02 | 신청 내역 조회, 관리자 승인 화면 후보 |
| 운영 모니터링 | E01~E05 | 대시보드, 실행 이력, 활용 현황 화면 후보 |
| 비정형 정보 관리 | F01~F02 | 문서 파싱 품질관리, 비정형 게시정보 관리 화면 후보 |
| 공통/관리 | G01~G02 | 사용자/권한 관리, 공지사항 화면 후보 |

## 템플릿 유형

| 템플릿 유형 | 의미 |
|---|---|
| `dashboard` | 여러 지표와 상태를 요약해서 보여주는 화면 |
| `search-list` | 조건 검색과 목록 조회가 중심인 화면 |
| `detail-view` | 특정 대상의 상세 정보를 읽는 화면 |
| `request-form` | 등록, 수정, 신청 입력이 중심인 화면 |
| `approval-workflow` | 승인/반려 판단이 중심인 화면 |
| `mapping-board` | 두 대상의 연결 관계를 조회하거나 조정하는 화면 |
| `quality-review` | 오류, 품질, 검수 상태를 확인하는 화면 |
| `admin-management` | 사용자, 권한, 공지 등 운영 관리 화면 |

## 개발 참고화면 구현 방안

### 유효성 판단

현재 템플릿 구조는 이 프로젝트의 설계 기준으로 유효하다.
이유는 화면 도메인과 구현 방식을 분리하고, 기능목록 엑셀의 A01~G02 항목을 업무 화면이 아니라 화면 형태 기준으로 먼저 분류할 수 있기 때문이다.

다만 템플릿 유형을 그대로 8개 업무 화면으로 만들면 안 된다.
템플릿은 실제 업무 화면이 아니라 개발자가 화면 구조를 참고하기 위한 샘플이어야 한다.

### 현재 프로젝트와의 적합성

| 템플릿 유형 | 현재 프로젝트 적합성 | 판단 |
|---|---|---|
| `dashboard` | `DashboardView.vue`가 이미 유사 구조를 가짐 | 기존 화면을 참고 가능 |
| `search-list` | `sample-list`가 검색, 목록, 페이징, 정렬 구조를 가짐 | 기존 화면을 참고 가능 |
| `request-form` | `sample-list`의 등록/수정 다이얼로그와 Vuelidate 구조를 참고 가능 | 기존 화면을 일부 활용 |
| `detail-view` | 읽기 전용 상세 화면 샘플은 부족함 | 참고 패널 추가 필요 |
| `approval-workflow` | 승인/반려 액션이 있는 상세 패턴은 부족함 | 참고 패널 추가 필요 |
| `mapping-board` | 관계 매핑형 UI는 현재 샘플에 없음 | 참고 패널 추가 필요 |
| `quality-review` | 오류/검수/상태 처리형 UI는 현재 샘플에 없음 | 참고 패널 추가 필요 |
| `admin-management` | 목록 + 등록/수정 패턴으로 대부분 대응 가능 | 별도 화면보다 변형 예시로 충분 |

### 권장 구현 형태

개발 참고화면은 업무 화면과 섞지 않고 하나의 라우트로 격리한다.

```text
frontend/src/features/template-reference/
  pages/TemplateReferencePage.vue
  template-catalog.ts
  components/
    DashboardTemplate.vue
    SearchListTemplate.vue
    DetailViewTemplate.vue
    RequestFormTemplate.vue
    ApprovalWorkflowTemplate.vue
    MappingBoardTemplate.vue
    QualityReviewTemplate.vue
    AdminManagementTemplate.vue
```

라우트는 하나만 둔다.

```text
/template-reference
```

화면 내부는 `ElTabs` 또는 좌측 메뉴로 템플릿 유형을 전환한다.
이렇게 하면 메뉴가 늘어나지 않고, 개발자가 한 화면에서 모든 패턴을 비교할 수 있다.

### 구현 원칙

- 실제 업무 화면 코드를 만들지 않는다.
- A01~G02 업무 코드를 직접 라우트로 만들지 않는다.
- 백엔드 API를 호출하지 않고 로컬 mock 데이터만 사용한다.
- 기존 공통 컴포넌트인 `GridPageLayout`, `SearchPanel`, `BaseGrid`, `AuthButton`, Vuelidate helper를 우선 사용한다.
- 템플릿 참고화면에서 반복이 확인된 구조만 나중에 공통 컴포넌트로 추출한다.
- 권한은 `TEMPLATE_READ` 같은 개발 참고용 권한 하나로만 둔다.
- 운영 배포 시 메뉴 노출 여부는 별도 환경 기준으로 제어한다.

### 구현 순서

| 순서 | 작업 | 결과 |
|---|---|---|
| 1 | `/template-reference` 라우트와 권한 코드 추가 | 개발 참고화면 진입점 생성 |
| 2 | `TemplateReferencePage.vue`에 템플릿 탭 구성 | 8개 템플릿 유형을 한 화면에서 전환 |
| 3 | `dashboard`, `search-list`, `request-form`은 기존 화면 구조를 축약해 반영 | 기존 샘플 재사용 |
| 4 | `detail-view`, `approval-workflow`, `mapping-board`, `quality-review` 참고 패널 추가 | 현재 부족한 패턴 보완 |
| 5 | `admin-management`는 목록 + 등록/수정 + 상태 토글 변형으로 구성 | 관리형 화면 기준 확보 |
| 6 | 빌드와 단위 테스트 확인 | 템플릿 참고화면 안정성 확인 |

### 완료 기준

- `/template-reference`에서 8개 템플릿 유형을 모두 확인할 수 있다.
- 화면은 로컬 mock 데이터만 사용한다.
- 업무 도메인명, 실제 API, 실제 테이블, 외부 연계 표현을 포함하지 않는다.
- 기존 `dashboard`, `sample-list`, `sample-list-jpa`와 역할이 겹치지 않는다.
- 실제 업무 화면 개발 시 어떤 템플릿을 참고해야 하는지 판단할 수 있다.

| 대상 | 현재 판단 | 비고 |
|---|---|---|
