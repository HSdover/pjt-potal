1. **각 산출물의 목차/표 구조 템플릿** — 빈 문서를 만들기 전에 형식부터 잠금
2. **미결 의사결정의 결정 기준(decision criteria)** — "무엇을 답해야 하나"가 아니라 "어떻게 판단하나"
3. **일자별·세션별 실행 계획** — 누가 언제 무엇을 어디까지
4. **검증 게이트의 객관적 통과 조건** — "정리되었다"가 아니라 "파일 X에 Y가 있다"

## 1. 코딩·문서 컨벤션 (원칙 보강)

원본 §1 원칙에 더해 PR 단위에서 강제할 컨벤션:

- 문서 파일은 모두 `docs/governance/` 하위. 회의록·draft도 같은 디렉터리에서 `_draft_` 접두사로 구분.
- 한국어 문서의 표는 모두 markdown 표(`|`), 코드 블록은 ```` ```text ```` 또는 ```` ```sql ```` 등 명시.
- 한 문서 내에서 RFP 코드는 항상 `REQ-070` 형식(하이픈, 3자리 zero-pad).
- 기능 코드는 `A01`~`G02` 형식 그대로 사용. 메뉴명은 기능목록 v0.3 그대로 인용.
- DTO/엔티티 명명: `snake_case` 컬럼, `PascalCase` 자바 클래스, `camelCase` 자바 필드.
- 외부 시스템 어댑터 패키지 prefix: `integration.<system>` (예: `integration.dataiku`).
- 데모 코드 prefix: `_template_` 또는 패키지 `com.example.governanceportal._template.<feature>`.

## 8. 통합본과 본 문서의 우선순위 정리

원본 통합본(보수, 게이트 중심) = **무엇을 만들지 + 무엇을 하지 말지**의 정합성 기준
본 문서(실행 디테일) = **각 산출물의 빈 페이지를 어떻게 채울지**의 기준

작업 순서는 원본 §8 순서를 그대로 따른다. 본 문서는 각 작업의 첫 30분에 "이 산출물의 골격을 §2 또는 §3 템플릿대로 잠그기"를 추가한다.

검증 흐름:

1. 작업 시작 → 본 문서 §2/§3 템플릿 복사로 빈 문서 생성
2. 작업 종료 → §2/§3 "객관적 통과 조건" 자가 점검
3. 머지 직전 → §6 PR 템플릿 8개 체크 완료
4. 6/5 진입 게이트 → 원본 통합본 §7 + 본 문서 §5 결정표 모두 완료
