# 작업 방향

작성일: 2026-05-19

## 현재 기준

현재 단계에서는 프론트엔드, 백엔드, DB 로직을 고려하지 않는다.
프로젝트가 실제로 시작되면 그때 기능 구현, API, 테이블, 권한, 연계 로직을 설계하고 개발한다.

지금 원하는 기능적 작업은 **기능목록 엑셀 기준으로 화면 도메인을 나누고, 각 화면을 템플릿 유형으로 분류하는 것**까지다.

| `docs/governance/screen-domain-template.md` | 기능목록 엑셀 기준 화면 도메인/템플릿 분류 |

## 지금 할 일

1. 기능목록 엑셀의 A01~G02 항목을 화면 후보로 정리한다.
2. 화면 후보를 상위 도메인으로 묶는다.
3. 각 화면 후보를 템플릿 유형으로 분류한다.
4. 하나의 화면으로 합칠 항목과 나중에 분리할 항목을 표시한다.
5. 기존 sample, sample-jpa, dashboard는 개발 참고용 템플릿으로만 본다.

## 지금 하지 않을 일

- 업무 화면 실제 구현
- 화면별 상세 기능 설계
- API, DTO, 테이블, ERD 설계
- 권한, 인증, 감사, 실패 처리 설계
- 외부 시스템 호출 방식 결정
- 메시지 큐, DLQ, webhook 처리 구조 설계
- 프론트엔드 컴포넌트 구조 확정
- 백엔드 패키지 구조 확정
- DB 저장 범위 결정
- 내부망 형상관리 전략 상세 설계
- Oracle Linux 운영 배포 절차 상세 설계

## 사용자 검토 체크리스트

| 확인 항목 | 판단 기준 |
|---|---|
| 도메인명 | 현업이 이해하는 메뉴 그룹명인가 |
| 화면명 | 기능목록 엑셀의 의도를 유지하는가 |
| 템플릿 유형 | 화면의 주 사용 방식과 맞는가 |
| 통합 가능성 | 비슷한 항목을 하나의 화면/탭으로 묶을 수 있는가 |
| 분리 필요성 | 같은 도메인 안에서도 사용 목적이 달라 별도 화면이 필요한가 |
| 보류 항목 | 엑셀만으로 판단하기 어려운 항목인가 |

## 다음 요청 순서

1. 기능목록 엑셀 기준 화면 도메인/템플릿 표를 보완한다.
2. 각 화면 후보를 독립 화면/탭/섹션 후보로 다시 나눈다.
3. 화면명과 메뉴명을 현업 용어 기준으로 다듬는다.
4. 프로젝트 투입 후 내부망 형상관리 전략을 상세화한다.
5. 프로젝트 투입 후 OCI Oracle Linux 운영 배포 전략을 상세화한다.

상세 화면 스펙 작성과 코드 구현은 프로젝트가 실제로 시작된 뒤 진행한다.
