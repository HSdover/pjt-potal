# 금주 작업사항 - 코덱스 문서

작성일: 2026-05-18

이 문서는 삼성증권 거버넌스포털 관련 문서와 현재 프로젝트를 비교한 결과를 바탕으로, 금주에 진행할 작업 방향을 정리한다.
현재 프로젝트는 샘플 데이터 기반 포털과 개발 템플릿 단계이므로, 이번 주 목표는 실제 업무 기능을 모두 구현하는 것이 아니라 외부 연계형 거버넌스 포털로 확장할 수 있는 기준 골격을 잡는 것이다.

## 이번 주 목표

샘플 포털을 유지하되, 앞으로의 개발 방향을 다음 구조로 전환한다.

```text
샘플 화면/샘플 DB 중심
-> 기능목록 기준 도메인/메뉴 구조
-> 외부 REST 연계 인터페이스와 Mock Adapter
-> 권한/감사/실패 처리 기준
-> 비정형/파서 Webhook 수신과 재처리 구조
```

## 작업 원칙

- OCI Data Catalog, ADB, Dataiku, GenON, Anyflow, 파서 솔루션의 고유 기능을 포털에 중복 구현하지 않는다.
- 포털은 통합 조회, 신청/승인 연결, 상태 모니터링, 감사/이력, 운영 편의 기능에 집중한다.
- 실제 외부 시스템이 준비되지 않아도 개발이 가능하도록 interface와 mock adapter를 먼저 둔다.
- 운영 데이터 원본을 포털 DB에 그대로 저장하지 않는다. 포털 DB에는 관리 메타, 연계 상태, 감사 로그, 동기화 작업 이력, 오류/재처리 정보를 중심으로 저장한다.
- RabbitMQ 같은 메시지 브로커는 바로 도입하지 않고, 먼저 DB 기반 작업 큐와 DLQ 개념을 설계한다. 처리량과 비동기 요구가 커질 때 RabbitMQ 또는 OCI Queue로 확장한다.

## 1. 기능목록 기준 메뉴/도메인 재정의

우선순위: 높음

현재 라우트는 `대시보드`, `메타데이터`, `원천 샘플`, `리니지`, `sample-list`, `sample-list-jpa` 중심이다.
문서 기준 메뉴는 아래 구조가 되어야 한다.

| 영역 | 목표 메뉴 |
|---|---|
| 메타 관리 | 통합 메타 검색, 데이터 상세 조회, 메타데이터 등록/수정 |
| AI 데이터 권한 관리 | Agent-데이터셋 매핑 조회, 접근 요청/승인 |
| 파이프라인 관리 | Dataiku 파이프라인 목록, 상용배포 신청 |
| 신청서 승인 | 신청 내역 조회, 관리자 승인/반려 |
| 운영 모니터링 | 통합 대시보드, 파이프라인 실행 현황, Agent 활용 현황 |
| 비정형 정보 관리 | 문서 파싱 결과 품질관리, 파싱 오류 관리, 비정형 게시정보 관리 |
| 공통/어드민 | 사용자/권한 관리, 보관주기 조회, 이슈/알림 관리 |

금주 작업:

- `sample-list`, `sample-list-jpa`는 개발 샘플로 분리할지, 메뉴에서 숨길지 결정한다.
- 라우터 `meta` 기준으로 실제 업무 메뉴 후보를 정의한다.
- 각 메뉴별로 필요한 외부 시스템과 내부 저장 테이블 후보를 정리한다.

완료 기준:

- 메뉴 구조와 라우트 후보가 문서로 정리된다.
- 샘플 화면과 실제 업무 화면의 역할이 분리된다.

## 2. 외부 REST 연계 인터페이스와 Mock Adapter 설계

우선순위: 높음

대부분의 데이터는 OCI와 외부 모듈에서 REST API로 받아올 예정이므로, 실제 연계보다 먼저 연계 경계를 고정해야 한다.

대상 연계:

| 외부 시스템 | 주요 용도 |
|---|---|
| 데이터포털 | 정형/비정형 메타데이터, 데이터 상세 속성 |
| OCI Data Catalog / ADB / ORDS | 카탈로그, Golden Layer, 접근 제어, 가상화 정보 |
| Dataiku | 파이프라인 목록, 실행 현황, 리니지 상세 진입 |
| GenON | AI Agent 정보, Agent-데이터셋 매핑 |
| Anyflow | 접근 권한 신청, 파이프라인 배포 신청, 승인/반려 |
| 파서 솔루션 | 문서 파싱 결과, 오류, 품질 점수 |
| WhaTap 또는 표준 모니터링 | 운영 지표, 장애/성능 정보 |

금주 작업:

- `backend`에 외부 연계용 client interface 패키지 기준을 잡는다.
- 실제 호출 구현 대신 mock adapter와 샘플 response DTO를 둔다.
- timeout, 오류 변환, retry 대상 여부 같은 공통 정책을 문서화한다.

완료 기준:

- 외부 시스템별 client interface 이름과 책임이 정해진다.
- 실제 API가 없어도 화면/API 개발을 계속할 수 있는 mock contract가 생긴다.

## 3. 인증/권한 기준 정리

우선순위: 높음

현재 백엔드는 `/api/**`를 모두 허용하고, 프론트는 샘플 권한 목록으로만 제어한다.
문서 기준으로는 IAM/Knox, OIDC/JWT, RBAC/ABAC, 관리자 접근 통제가 필요하다.

금주 작업:

- 개발용 보안 정책과 운영용 보안 정책을 분리한다.
- 라우터 `meta.auth`와 백엔드 authority naming 규칙을 맞춘다.
- RBAC는 우선 역할 기반으로 잡고, ABAC는 Agent/데이터셋/보안등급/부서 조건을 확장 후보로 둔다.
- 관리자 메뉴와 API는 별도 권한 코드로 분리한다.

완료 기준:

- `permitAll` 상태가 운영 기준으로 오해되지 않도록 문서와 profile 기준이 분리된다.
- 메뉴 권한, 버튼 권한, API 권한의 이름 규칙이 정해진다.

## 4. 감사/이력 및 실패 처리 구조 설계

우선순위: 높음

문서에는 API, SQL, 접근 이벤트를 기본 감사 로그로 수집하고, 사용자별/Agent별 활용 이력을 추적해야 한다는 요구가 있다.
외부 REST 연계가 많아질수록 실패 이력과 재처리 구조도 필요하다.

금주 작업:

- 감사 로그 후보 테이블을 설계한다.
  - API 접근 로그
  - 데이터 조회 이력
  - Agent 데이터 활용 이력
  - 승인/반려 이력
- 외부 동기화 작업 테이블을 설계한다.
  - 작업 상태: `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`, `DEAD`
  - 재시도 횟수, 마지막 오류, 요청 payload, 응답 요약
- DLQ는 초기에 별도 브로커가 아니라 `DEAD` 상태 또는 dead-letter 테이블로 처리한다.

완료 기준:

- 어떤 이벤트를 감사 로그로 남길지 정리된다.
- 외부 API 실패가 유실되지 않고 재처리 가능한 구조로 정리된다.

## 5. 비정형/파서 Webhook 수신 구조 설계

우선순위: 중간

문서에는 파서 솔루션 결과를 받기 위한 Webhook/Callback REST API와 파싱 오류 재처리 구조가 필요하다고 되어 있다.

금주 작업:

- 파서 결과 수신 API 초안을 정의한다.
  - 예: `POST /api/webhooks/parser/results`
- 요청 인증 방식을 결정한다.
  - 예: API Key, HMAC signature, 내부망 allowlist
- Webhook 수신 후 즉시 처리하지 않고 inbox/job에 저장한 뒤 200 OK를 반환하는 구조로 잡는다.
- 파싱 오류, 품질 점수, 원천 문서 식별자, 재처리 상태 필드를 정리한다.

완료 기준:

- Webhook을 받을 엔드포인트와 payload 초안이 정리된다.
- 파싱 실패가 메인 처리 흐름을 막지 않는 구조가 정리된다.

## 6. 대시보드/모니터링 방향 재정의

우선순위: 중간

현재 대시보드는 HIRA 샘플 상태를 보여준다.
문서 기준 대시보드는 파이프라인 실패율, 사용량 추이, Agent 활용 현황, 품질 저하/오류 상태를 보여줘야 한다.

금주 작업:

- 대시보드 지표 후보를 실제 문서 요구 기준으로 재정의한다.
- 현재 샘플 대시보드는 개발용으로 유지하고, 운영 대시보드 DTO를 별도 설계한다.
- Dataiku, GenON, WhaTap, 내부 audit log 중 어떤 지표가 어느 출처에서 오는지 매핑한다.

완료 기준:

- 운영 대시보드의 지표명, 출처, 갱신 방식, 저장 여부가 정리된다.

## 이번 주에 하지 않을 것

- 실제 OCI, Dataiku, GenON, Anyflow 운영 API 연결
- RabbitMQ 도입
- 완전한 SSO/OIDC 운영 인증 구현
- 모든 업무 화면 일괄 구현
- 디자인/퍼블리싱 고도화

위 항목은 기반 구조가 정리된 뒤 단계적으로 진행한다.

## Codex 작업 요청 단위

다음 작업은 Codex에 개별 요청으로 나눠 진행하는 것이 좋다.

1. 기능목록 기준으로 라우터/메뉴 구조 초안을 문서화하고, 샘플 메뉴를 개발용으로 분리해줘.
2. 외부 시스템 연계 client interface와 mock adapter 패키지 구조를 만들어줘.
3. IAM/OIDC 운영 전환을 고려해서 개발/운영 SecurityConfig 분리 방향을 잡아줘.
4. 감사 로그와 외부 동기화 작업 테이블 초안을 schema 문서 또는 SQL로 만들어줘.
5. 파서 Webhook 수신 API와 inbox/job 처리 흐름을 설계해줘.
6. 운영 대시보드 지표 DTO와 mock API를 문서 요구사항 기준으로 재정의해줘.

## 판단 기준

이번 주 작업이 끝났을 때 아래 질문에 답할 수 있어야 한다.

- 이 화면은 어떤 문서 요구사항에 대응하는가?
- 데이터는 포털 DB에서 관리하는가, 외부 시스템에서 조회하는가?
- 외부 API가 실패하면 어디에 기록되고 어떻게 재처리되는가?
- 사용자의 접근과 데이터 조회는 어디에 감사 로그로 남는가?
- 샘플 구현과 운영 구현의 경계가 명확한가?
