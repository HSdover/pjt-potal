# 금주 작업사항 - 통합본

작성일: 2026-05-18

기준 문서:

- `개선사항/금주작업사항-코덱스문서.md`
- `개선사항/금주작업사항-클로드문서.md`
- `logs/doc-extract/` 5종 문서

이 문서는 코덱스 문서의 보수적인 방향성과 클로드 문서의 실행 계획을 통합한 금주 작업 기준이다.
핵심 방향은 **샘플 포털을 바로 대규모 업무 화면으로 확장하지 않고, RFP 기능 매핑과 설계 게이트를 먼저 통과한 뒤 생성/구현 범위를 확정하는 것**이다.

## 1. 작업 원칙

- 포털은 OCI Data Catalog, ADB, Dataiku, GenON, Anyflow, 파서 솔루션의 고유 기능을 중복 구현하지 않는다.
- 포털은 통합 조회, 신청/승인 연결, 운영 모니터링, 감사/이력, 외부 연계 실패 관리에 집중한다.
- 운영 데이터 원본은 포털 DB에 그대로 저장하지 않는다. 포털 DB에는 관리 메타, 연계 상태, 감사 로그, 작업 이력, 오류/재처리 정보를 중심으로 저장한다.
- 실제 외부 시스템이 준비되지 않아도 개발할 수 있도록 외부 연계는 `interface + mock adapter`를 먼저 둔다.
- RabbitMQ 같은 브로커는 바로 도입하지 않는다. 먼저 DB 기반 작업 큐와 DLQ 개념을 설계하고, 처리량이 커질 때 RabbitMQ 또는 OCI Queue로 확장한다.
- 데모와 RFP 업무 기능은 패키지, 라우터, 권한 코드에서 명확히 분리한다.
- API는 Springdoc에 노출되는 계약을 우선한다. Swagger에 보이지 않는 API는 완료로 보지 않는다.
- 새 의존성 추가 시 오프라인 빌드 영향이 있는지 확인한다.

## 2. 금주 P0 산출물

P0는 금주에 끝내야 하는 설계 게이트다.
코드를 많이 만드는 것보다, 6월 본 구현에 들어갈 때 충돌하지 않도록 **범위와 경계를 잠그는 것**이 목표다.

### P0-1. 데모 기능과 RFP 기능 분리 매핑

산출물:

- `docs/governance/domain-mapping.md`

내용:

- 현재 화면과 API를 전수한다.
  - 대시보드
  - 메타데이터
  - 원천 샘플
  - 리니지
  - `sample-list`
  - `sample-list-jpa`
- 기능목록 A01~G02와 현재 구현을 매핑한다.
- 각 항목을 아래 셋 중 하나로 분류한다.
  - RFP 업무 기능으로 승격
  - 개발 템플릿으로 유지
  - 제거 또는 메뉴 숨김
- `sample`, `sample-list-jpa`, HIRA 기반 샘플 화면을 운영 라우터에 둘지 결정한다.

검증 기준:

- 신규 생성이 필요한 업무 도메인 목록이 명확해야 한다.
- 샘플 화면과 운영 화면의 경계가 문서로 설명되어야 한다.

### P0-2. 거버넌스 도메인 ERD 초안

산출물:

- `docs/governance/erd-v0.1.md`
- `docs/governance/schema-governance-draft.sql`

주의:

- 이번 주에는 설계 초안으로 둔다.
- 바로 `backend/src/main/resources/schema.sql`에 병합하지 않는다.
- H2와 Oracle ADB 26ai의 차이를 고려해서, 실행 SQL과 설계 SQL을 분리한다.

포함할 핵심 엔티티:

| 영역 | 후보 엔티티 |
|---|---|
| 메타 관리 | `metadata_catalog`, `metadata_attribute` |
| Agent 권한 | `agent`, `dataset`, `agent_dataset_mapping` |
| 신청/승인 | `access_request`, `access_request_history`, `approval_log` |
| 파이프라인 | `pipeline`, `pipeline_run` |
| 비정형/파서 | `unstructured_doc`, `parser_result`, `parser_quality_metric` |
| 감사/이력 | `audit_log`, `data_access_log`, `agent_usage_log` |
| 외부 연계 실패 | `sync_job`, `dead_letter` |
| 데이터포털 양방향 | `data_portal_sync_inbox`, `data_portal_sync_outbox` |
| 사용자/권한 | `user_account`, `user_role`, `permission`, `user_lifecycle_event` |
| 공통 | `notification`, `data_retention_policy` |

검증 기준:

- A01~G02 메뉴가 어떤 엔티티 또는 외부 시스템을 사용하는지 설명되어야 한다.
- 포털 DB가 원본 데이터를 과도하게 소유하지 않는 구조여야 한다.

### P0-3. 16개 화면 spec 초안과 화면 유형 분류

산출물:

- `docs/governance/screen-spec-plan.md`
- `tools/generator/pages/*` spec 초안은 확정된 항목부터 작성

중요한 게이트:

- 16개 화면을 모두 `search-grid`로 일괄 생성하지 않는다.
- 먼저 화면 유형을 분류한다.

화면 유형 후보:

| 유형 | 적용 후보 |
|---|---|
| `search-grid` | 통합 목록, 실행 목록, 신청 내역 |
| `search-grid-detail` | 데이터 상세, Agent 매핑 상세, 파싱 결과 상세 |
| `approval-workflow` | 접근 요청, 상용배포 신청, 승인/반려 |
| `dashboard` | 통합 운영 대시보드, Agent 활용 현황 |
| `external-link` | Dataiku 상세 진입, 외부 카탈로그 진입 |
| `admin-form` | 사용자/권한 관리, 보관주기 기준 조회 |

금주 범위:

- 16개 화면의 spec 초안을 만든다.
- 실제 생성은 대표 2~3개 화면만 먼저 검증한다.
- 생성기 확장은 필요한 유형이 확정된 뒤 진행한다.

검증 기준:

- 각 화면이 RFP 코드, 메뉴명, 외부 연계, 화면 유형, 권한 코드를 가진다.
- 생성기 현재 한계(`search-grid`만 지원)를 문서에서 명확히 표시한다.

### P0-4. API-First 계약 초안

산출물:

- `docs/governance/api-contract-plan.md`
- 대표 업무 API 2~3개 컨트롤러 스켈레톤

권장 대표 API:

- A01 통합 메타 검색
- B01 Agent-데이터셋 매핑 조회
- E02 파이프라인 실행 목록 조회

작업 방향:

- 모든 API는 RFP 코드와 화면 spec을 연결한다.
- `ListRequest`, `ListResponse` 공통 계약을 유지한다.
- 컨트롤러에는 `@Operation`을 붙인다.
- `@PreAuthorize`는 자리만 잡되, 실제 보안 적용은 P1에서 한다.

검증 기준:

- Swagger에서 대표 API가 보인다.
- 전체 16개 API를 바로 만들지 말고, 대표 API로 패턴이 맞는지 검증한다.

## 3. P1 착수 과제

P1은 금주에 착수하고 다음 주까지 완료하는 과제다.
P0 산출물이 확정된 뒤 코드 구현 범위를 넓힌다.

### P1-1. 외부 연계 인터페이스와 Mock Adapter

산출물:

- `docs/governance/integration-profiles.md`
- `backend` 외부 연계 패키지 구조

대상:

| 외부 시스템 | 주요 용도 |
|---|---|
| 데이터포털 | 정형/비정형 메타데이터, 데이터 상세 속성 |
| OCI Data Catalog / ADB / ORDS | 카탈로그, Golden Layer, 접근 제어, 가상화 정보 |
| Dataiku | 파이프라인 목록, 실행 현황, 리니지 상세 진입 |
| GenON | AI Agent 정보, Agent-데이터셋 매핑 |
| Anyflow | 접근 권한 신청, 파이프라인 배포 신청, 승인/반려 |
| 파서 솔루션 | 문서 파싱 결과, 오류, 품질 점수 |
| WhaTap | 운영 지표, 장애/성능 정보 |
| IAM/Knox | 사용자, 조직, 역할, 권한 회수 이벤트 |

작업 방향:

- 외부 API raw client를 화면 서비스에 직접 노출하지 않는다.
- `XxxClient` interface와 `MockXxxClient`를 먼저 둔다.
- timeout, retry 여부, 오류 변환, 인증 헤더 정책을 공통화한다.

검증 기준:

- 실제 외부 시스템 없이 mock 응답으로 대표 화면이 동작해야 한다.

### P1-2. 인증/권한 PoC

산출물:

- `docs/governance/auth-strategy.md`
- 개발용/운영용 보안 profile 분리 방향

작업 방향:

- 현재 `/api/** permitAll` 상태를 운영 기준으로 오해하지 않게 분리한다.
- 로컬 PoC는 Keycloak 또는 모의 JWT 중 하나로 결정한다.
- `GET /api/me`로 프론트가 사용자와 권한을 받는 구조를 잡는다.
- `@PreAuthorize`를 쓰려면 `@EnableMethodSecurity`를 포함한다.

검증 기준:

- 권한 없는 토큰은 401/403이 나와야 한다.
- 권한 있는 토큰은 대표 API를 호출할 수 있어야 한다.

### P1-3. 감사/이력과 외부 실패 처리

산출물:

- `docs/governance/audit-and-sync-job.md`
- 감사 로그/작업 큐 DDL 초안

포함 내용:

- `audit_log`: API 접근, 주요 업무 액션, 승인/반려, 데이터 조회
- `data_access_log`: 사용자별 데이터 조회 이력
- `agent_usage_log`: Agent별 데이터 활용 이력
- `sync_job`: 외부 REST 연계 작업 상태
- `dead_letter`: 재시도 초과 또는 처리 불가 payload 보관

작업 방향:

- 처음부터 RabbitMQ를 넣지 않는다.
- DB 기반 `PENDING/RUNNING/SUCCESS/FAILED/DEAD` 상태 전이를 설계한다.
- 재처리 API를 설계한다.

검증 기준:

- 외부 API 실패가 유실되지 않고 재처리 가능한 경로를 가진다.

### P1-4. RFP Trace 매트릭스

산출물:

- `docs/governance/rfp-trace.md`

작업 방향:

- REQ-070~REQ-092를 화면, API, 도메인, 외부 시스템과 연결한다.
- spec과 Swagger description에도 RFP 코드를 넣는다.

검증 기준:

- 어떤 요구사항이 어느 화면/API에서 대응되는지 추적 가능해야 한다.

## 4. P2 이월 과제

P2는 이번 주에 바로 구현하지 않고, 착수 시점만 정한다.

| 과제 | 착수 조건 |
|---|---|
| ECharts 기반 운영 대시보드 | 운영 지표 DTO와 출처가 확정된 뒤 |
| Spring Batch 통계 사전집계 | `pipeline_run`과 마트 테이블 설계 확정 뒤 |
| 파서 Webhook 수신 API | parser payload와 인증 방식 확정 뒤 |
| RabbitMQ 또는 OCI Queue 도입 | DB 작업 큐 한계가 확인된 뒤 |
| SonarQube/SAST/CI 품질 게이트 | 핵심 도메인 패키지 구조가 안정된 뒤 |
| 리니지 전략 결정 | Dataiku 위임 여부와 SSO 진입 방식 확정 뒤 |
| Knox/2FA 전략 | 보안팀 연동 가능 시점 확인 뒤 |

## 5. 이번 주에 하지 않을 것

- 실제 OCI, Dataiku, GenON, Anyflow, WhaTap 운영 API 연결
- RabbitMQ 도입
- 완전한 SSO/OIDC 운영 인증 구현
- 16개 화면을 전부 실제 화면으로 생성
- 16개 컨트롤러를 검증 없이 일괄 구현
- 운영 DB 스키마를 기존 `schema.sql`에 바로 병합
- 디자인/퍼블리싱 고도화

## 6. 미결 의사결정

이번 주 안에 답해야 하는 항목:

- IAM/Knox 연동 PoC가 가능한가, 아니면 모의 JWT로 갈 것인가?
- 리니지는 Vue Flow 자체 구현을 유지할 것인가, Dataiku 상세 화면 위임 + SSO 링크로 갈 것인가?
- HIRA 기반 샘플 화면은 운영 라우터에서 제거할 것인가, `_template` 영역으로 격리할 것인가?
- DDL은 Oracle ADB 기준으로 먼저 잡을 것인가, H2 호환 기준으로 먼저 잡을 것인가?
- WhaTap 연동은 금주 설계 범위에 넣을 것인가?
- Webhook 인증은 API Key, HMAC, 내부망 allowlist 중 무엇으로 갈 것인가?
- 비정형 원천 5종은 통합 adapter로 볼 것인가, 원천별 adapter로 분리할 것인가?
- 데이터포털 양방향 연계에서 충돌 해결 기준은 누가 소유하는가?

## 7. 검증 기준

금주 작업이 끝났을 때 아래 질문에 답할 수 있어야 한다.

1. 이 화면/API는 어떤 RFP 요건에 대응하는가?
2. 이 데이터는 포털 DB에서 관리하는가, 외부 시스템에서 조회하는가?
3. 포털 DB가 원본 데이터를 과도하게 소유하지 않는가?
4. 외부 API가 실패하면 어디에 기록되고 어떻게 재처리되는가?
5. 사용자의 접근과 데이터 조회는 어디에 감사 로그로 남는가?
6. 샘플 구현과 운영 구현의 경계가 명확한가?
7. 해당 기능은 mock adapter로 외부 시스템 없이 개발 가능한가?
8. Swagger 또는 문서에서 API 계약을 확인할 수 있는가?

## 8. 추천 작업 요청 순서

Codex 또는 Claude에 작업을 나눠 요청할 때는 아래 순서를 권장한다.

1. `docs/governance/domain-mapping.md`를 만들고 데모/RFP 기능을 분리해줘.
2. `docs/governance/erd-v0.1.md`와 `schema-governance-draft.sql` 초안을 만들어줘.
3. 16개 화면을 화면 유형별로 분류하고 `screen-spec-plan.md`를 만들어줘.
4. 대표 API 2~3개만 Springdoc 스켈레톤으로 만들어 패턴을 검증해줘.
5. 외부 연계 client interface와 mock adapter 패키지 구조를 만들어줘.
6. 인증/권한 PoC 문서를 만들고 `permitAll` 운영 오해를 막는 profile 전략을 잡아줘.
7. 감사 로그와 `sync_job/dead_letter` 재처리 구조를 설계해줘.
8. RFP trace 매트릭스를 만들어 요구사항 누락을 확인해줘.

## 9. 참고

- 코덱스 문서는 작업 원칙과 과잉 구현 방지에 강점이 있다.
- 클로드 문서는 산출물, P0/P1/P2, 검증 기준을 구체화한 점이 강점이다.
- 통합본은 코덱스의 안전한 순서와 클로드의 실행 단위를 결합하되, 16개 화면/컨트롤러 일괄 생성은 설계 게이트 이후로 낮췄다.
