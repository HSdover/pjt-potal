**Why**: 현재 `metadata/sourcesample/lineage/sample/samplejpa`는 전부 HIRA·템플릿 데모. 어떤 것을 RFP A01~G02 중 무엇으로 "승격"하고 어떤 것을 정리할지 정하지 않으면 6/8부터 화면이 충돌한다.

- 결정 사항: `sample`, `sample-list-jpa` 등 데모 화면을 **production 라우터에 남길지** / **`/_template` 네임스페이스로 격리할지** 명시

**작업 방법**:
1. `frontend/src/router/index.ts`, `backend/src/main/java/.../**/api/*Controller.java` 전수.

3. 매핑 안 되는 것은 "데모"로 명시하고 정리 계획(삭제/이름변경/`_template/` 이동)을 작성.

**검증**: 표 2의 신규 도메인 목록이 P0-2(ERD 초안)와 100% 일치해야 함.

**Why**: 현재 `schema.sql`은 데모 5개 테이블뿐. 핵심 도메인이 빠지면 16개 화면 spec도 못 짠다.

- 최소 엔티티(이름은 조정 가능):

**검증**:

- 데모 5개 테이블(`sample`/`sample_jpa`/`hira_*`/`metadata_dataset`/`lineage_flow`)과 충돌하지 않거나, 충돌 시 P0-1에서 정한 정리 계획에 따라 처리.

### P0-3. 16개 기능 화면 spec 파일 일괄 작성
**Why**: 분석/설계 종료(6/5) 전에 라우터·feature 스켈레톤이 잡혀야 6/8부터 화면 16개를 병렬 개발 가능. `tools/generator`가 이미 있으므로 spec만 쓰면 자동 생성됨.

**Deliverable**:
- `tools/generator/pages/{a01-meta-search,a02-meta-detail,...,g02-notice}.json` 총 16개
- 화면 1개당 `feature/type/title/description/auth/apiPath` 최소 필드 채움
- 권한 코드 표준: `<DOMAIN>_<READ|CREATE|UPDATE|DELETE|APPROVE>` (예: `META_READ`, `AGENT_MAPPING_APPROVE`)

**작업 방법**:
1. 기능목록 v0.3 16행을 그대로 spec 16개로 옮긴다.
2. 각 spec의 `type`은 현재 `search-grid`만 지원하므로 일단 전부 `search-grid`로 두고, `search-grid-detail`/`form`/`dashboard`가 필요한 화면은 spec에 `TODO_TYPE` 주석을 넣어둔다(생성기 확장은 다음 P 항목).
3. 생성 명령:
   ```powershell
   tools\generator\generate-feature.cmd
   ```
   (`generator.config.json`의 `spec` 값을 16번 바꾸지 말고, **batch 실행 스크립트** `tools/generator/generate-all.cmd`를 신규로 만든다 — 모든 spec을 순회 실행)
4. 권한 코드 16개 × CRUD를 `frontend/src/stores/auth.ts`의 `SAMPLE_PERMISSIONS`에 추가. 이름은 `SAMPLE_PERMISSIONS` → `APP_PERMISSIONS`로 리네임 권고.

**검증**: `npm.cmd run build` 통과 + 라우터에서 16개 메뉴 노출 + 권한 없는 사용자로 접근 시 `/forbidden` 도달.

### P0-4. API-First 계약 정의 (Springdoc 기반)
**Why**: 사전준비 PPT 원칙 1 = "UI용 API가 MCP/ORDS로 재노출". 컨트롤러 스펙이 먼저 잡혀야 프론트가 mock·실연동을 병렬 진행 가능.

**Deliverable**:
- 16개 화면용 컨트롤러 스켈레톤 16개 (`@RestController` + `@Operation` + Record DTO + `ListRequest/ListResponse` 공통 계약 사용)
- 모든 엔드포인트는 mock 데이터 반환으로 시작 (`Service`는 일단 `Mockxxx` 또는 메모리 List)
- `springdoc` 산출물 확인: `http://localhost:18080/swagger-ui.html`

**작업 방법**:
1. P0-1/P0-2/P0-3 결과에 맞춰 패키지 구조 잡기 — 예: `com.example.governanceportal.{meta,agent,pipeline,approval,monitoring,unstructured,admin}`.
2. 컨트롤러는 기존 `MetadataController` 패턴(`POST /search` + `ListRequest`) 그대로 채택.
3. 모든 DTO를 Java 21 `record`로(계획서 2.3 Record DTO 방침). Lombok 사용 최소화.
4. 권한 어노테이션 자리만 잡아둠: `@PreAuthorize("hasAuthority('META_READ')")` — 실제 SecurityConfig 실연동은 P1-2에서.

**검증**: Swagger UI에서 16개 화면용 endpoint 모두 호출 가능, mock 응답 정상.

### P1-2. 인증 PoC (1곳만, JWT decoder + SecurityConfig 실연동)
**Why**: 현재 `SecurityConfig`가 전부 `permitAll()` — 본 구현 들어가면 컨트롤러마다 권한 어노테이션이 무의미해진다. **IAM/Knox 정식 연동은 보안망 이관 후이지만, 구조는 지금 잡아야 한다.**

**Deliverable**:
- 옵션 A: 로컬 OIDC IdP(Keycloak Docker 1컨테이너)와 연동하는 PoC
- 옵션 B: IAM/Knox 연동이 어렵다면 "사내 JWT 발급 모의 서버"로 우회 — 어떤 옵션이든 **Resource Server 검증 흐름과 권한 매핑이 동작**해야 함
- `SecurityConfig`:
  - `/api/**` → `oauth2ResourceServer().jwt()` 적용
  - `/api/public/**`, `/swagger-ui/**`, `/actuator/health` 만 permitAll
  - `JwtAuthenticationConverter`로 JWT claim → `GrantedAuthority` 매핑 (`META_READ` 등)
- 프론트: `useAuthStore`의 하드코딩 permissions 제거 → 백엔드 `GET /api/me`에서 권한 수신

**작업 방법**:
1. 보안망 IAM/Knox 연동이 가능한지 인프라/보안팀 확인 → 가능하면 옵션 A, 아니면 옵션 B.
2. `application.yml`에 `spring.security.oauth2.resourceserver.jwt.issuer-uri` 추가, 프로파일로 분기.
3. PoC는 1개 화면(예: 메타 검색 A01)에 `@PreAuthorize` 적용해 권한 차단 동작 확인.

**검증**: 권한 없는 토큰으로 호출 시 401/403, 권한 있는 토큰은 정상 응답.

### P1-3. 횡단 관심사(AOP) 골격 — 감사 로그 + 이벤트 발행
**Why**: 사전준비 PPT 원칙 2(계층분리: 권한·감사·리니지는 AOP) / 원칙 3(Observability-by-Design). 컨트롤러마다 흩어지면 일정 안에 못 끝낸다.

**Deliverable**:
- `common/audit/AuditAspect.java`: `@AuditLogged` 어노테이션 + `audit_log` 테이블 INSERT
- `common/event/`: 신청서 상태 전환 이벤트(`AccessRequestSubmittedEvent`, `AccessRequestApprovedEvent` …) + `@EventListener` 1개 샘플(이메일 발송 자리만)
- Auditing 활성화: `@EnableJpaAuditing`, `AuditorAware<String>` 구현(JWT principal 기반)

**작업 방법**:
1. `audit_log` 테이블 스키마는 P0-2 ERD에 포함.
2. AOP는 컨트롤러 메서드 `@AuditLogged(action="META_SEARCH")` 형태로 선언적으로 사용.
3. 이벤트는 ApplicationEventPublisher 직접 사용, 트랜잭션 종료 후 처리(`@TransactionalEventListener(phase = AFTER_COMMIT)`).

**검증**: 화면 1개 호출 시 `audit_log`에 행 적재, 이벤트 1개 발행 시 listener 진입 로그.

## 작업 시 지켜야 할 원칙 (코드/리뷰 체크리스트)

1. **외부 시스템은 인터페이스 뒤에 숨긴다** — Mock 어댑터로 화면 개발 항상 가능해야 함.
2. **데모와 RFP는 절대 섞지 않는다** — 패키지·라우터·권한 코드 모두 분리. 데모는 `_template` 네임스페이스/주석.
3. **DTO는 Java 21 record** — Lombok `@Data` 지양(계획서 2.3).
4. **권한은 라우터·컨트롤러 양쪽** — 프론트만 막으면 우회됨. `@PreAuthorize`는 P0-4부터 자리만이라도 잡아둔다.
5. **JPA Auditing + AOP 감사**는 컨트롤러에 분기 코드 넣지 않는다.
6. **상태머신은 enum + 검증 메서드**로 — `if/else` 흩어 두지 않는다(계획서 2.2).
7. **오프라인 빌드 깨뜨리지 않기** — 새 의존성 추가 시 `scripts/prepare-offline-bundle.ps1` 재실행 필수.
8. **API-First**: Swagger에 안 보이면 만든 게 아니다. 컨트롤러에 `@Operation(summary=...)` 강제.
